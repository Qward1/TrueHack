## Слайд 1. Титульный

**Заголовок:**  
LocalScriptLua

**Подзаголовок:**  
Локальная AI-система для генерации, доработки, проверки и сохранения LowCode / workflow Lua-скриптов

**Текст на слайд:**
- Генерация Lua 5.5 по задаче на естественном языке

- Система ориентирована на рабочий инженерный цикл, а не только на “сгенерировать текст”.

## Слайд 2. Какую проблему решает проект

**Заголовок:**  
Проблема и цель

**Текст на слайд:**
- LowCode Lua-задачи часто описываются естественным языком
- Обычная LLM-генерация нестабильна по формату и логике
- Нужны прямые workflow-paths, без JsonPath и без “выдуманных” полей
- Нужен не только код, но и проверка, сохранение и объяснение результата

**Что сказать:**
- Главная проблема в том, что генерация workflow-скриптов — это не просто “написать Lua”.
- Нужно соблюдать контракт платформы: `wf.vars`, `wf.initVariables`, wrapper `lua{...}lua`, корректная работа с массивами, отсутствие консольного I/O и т.д.
- Поэтому проект построен как пайплайн со стадиями контроля качества.

## Слайд 3. Что умеет система

**Заголовок:**  
Функциональность проекта

**Текст на слайд:**
- Создаёт новый Lua-код
- Дорабатывает существующий код
- Отвечает на вопросы по текущему коду
- Проверяет код локально через Lua harness
- Делает fix-loop при ошибках
- Проверяет соответствие исходной задаче через LLM-verifier
- Сохраняет `.lua` и sidecar `jsonstring`

**Что сказать:**
- Это не один агент и не один endpoint.
- Система умеет работать как полноценный workflow-ассистент: генерация, refine, retry, inspect, explain.
- Отдельно важно, что код не только выдаётся в ответе, но и при необходимости сохраняется в целевой `.lua`-файл и sidecar-артефакт.

## Слайд 5. Структура проекта

**Заголовок:**  
Структура репозитория

**Текст на слайд:**
- `app.py` — UI, HTTP API, chat runtime, console режимы
- `src/graph/*` — LangGraph-пайплайн и маршрутизация
- `src/tools/*` — RAG, Lua validation, target/save, runtime helpers
- `src/core/*` — LLM adapter, state, логирование
- `src/agents/planner.py` — отдельный TaskPlanner
- `tests/*` — тесты пайплайна, planner, RAG, логирования
- `docker-compose.yml`, `Dockerfile` — развёртывание
- `openapi.yaml` — API-контракт

**Что сказать:**
- Проект разбит по слоям.
- `app.py` — это внешний слой взаимодействия.
- Вся предметная логика вынесена в пайплайн и инструменты, поэтому система расширяется без переписывания UI.

## Слайд 7. Полный граф пайплайна

**Заголовок:**  
LangGraph: полный граф выполнения

**Текст на слайд:**

```text
START
 ├─ route_from_start
 │   ├─ normal -> resolve_target
 │   └─ planner_followup -> plan_request
 │
resolve_target
 -> route_intent
    ├─ prepare -> plan_request
    └─ answer -> answer_question -> END

plan_request
 ├─ continue -> prepare_generation_context
 └─ clarify -> prepare_response -> END

prepare_generation_context
 ├─ generate -> generate_code -> validate_code
 ├─ refine -> refine_code -> validate_code
 └─ clarify -> prepare_response -> END

validate_code
 ├─ verify -> verify_requirements
 └─ fix_validation -> fix_validation_code -> validate_code

verify_requirements
 ├─ save -> save_code -> explain_solution -> prepare_response -> END
 └─ fix_verification -> fix_verification_code -> verify_requirements
```

**Что сказать:**
- Это центральный инженерный артефакт проекта.
- Здесь видно, что система умеет как генерацию, так и question/inspect-ветку.
- В графе есть отдельные циклы исправления после провала validation и verification.

## Слайд 8. Что происходит в начале пайплайна

**Заголовок:**  
Стартовые этапы: путь, интент, планирование

**Текст на слайд:**
- `resolve_target` определяет файл, папку или active target текущего чата
- `route_intent` через LLM решает: create / change / inspect / question / retry / general
- `plan_request` переформулирует задачу и при необходимости задаёт уточнения
- `prepare_generation_context` компилирует контекст: workflow paths, типы, операции, ожидания

**Что сказать:**
- Здесь система превращает сырое сообщение пользователя в инженерно пригодную постановку задачи.
- На этом этапе она понимает, будет ли это новый код, доработка, вопрос по коду или повторная проверка.
- Если задача неоднозначна, пользователь получает уточняющие вопросы, а не случайный код.

## Слайд 9. Генерация и доработка кода

**Заголовок:**  
Генерация и refine

**Текст на слайд:**
- `generate_code` создаёт новый Lua-скрипт
- `refine_code` изменяет уже существующий код
- Все промпты привязаны к LowCode-контракту
- Ответ нормализуется к строгому формату `lua{...}lua`
- При обрыве ответа есть continuation / retry логика

**Что сказать:**
- Важный момент: проект не доверяет raw-ответу модели.
- Даже корректный по смыслу код сначала проходит нормализацию формата.
- Для refine-ветки дополнительно сохраняются функции, чтобы модель не “теряла” существующую структуру.

## Слайд 10. Локальная валидация и два fix-loop

**Заголовок:**  
Контроль качества после генерации

**Текст на слайд:**
- `validate_code` запускает Lua-код в локальном harness
- Harness поднимает mock `wf.vars`, `wf.initVariables`, `_utils.array.*`
- `fix_validation_code` чинит runtime и syntax ошибки
- `verify_requirements` проверяет логику против исходной задачи
- `fix_verification_code` исправляет семантические ошибки

**Что сказать:**
- Это один из самых сильных блоков проекта.
- Здесь есть разделение между “код исполняется” и “код делает именно то, что просили”.
- За счёт этого система отсеивает правдоподобный, но неверный код.

**Место под скрин:**

## Слайд 11. RAG и выбор шаблона

**Заголовок:**  
Template RAG для CodeGenerator

**Текст на слайд:**
- Локальная KB: `lua_rag_templates_kb.jsonl`
- В базе сейчас `136` шаблонов
- Эмбедер: `qwen3-embedding:0.6b`
- Retrieval: `top_k = 5`
- `TemplateSelector` на `qwen2.5-coder:3b-instruct` выбирает один лучший шаблон
- В `CodeGenerator` инжектится только выбранный шаблон

**Что сказать:**
- Важно, что в кодогенератор не отправляются сразу все найденные шаблоны.
- Сначала retrieval находит кандидатов, потом отдельная модель-ранкер выбирает лучший.
- Это уменьшает шум в prompt и делает генерацию более точной.

## Слайд 13. Сохранение, артефакты и наблюдаемость

**Заголовок:**  
Артефакты и observability

**Текст на слайд:**
- Сохранение в `.lua` по явному пути или active target чата
- Параллельный sidecar `jsonstring`-артефакт
- SQLite хранит чаты, сообщения и состояние
- `runtime.jsonl` хранит технические события пайплайна
- `llm_prompts.jsonl` хранит вызовы моделей и prompt-аудит
- Есть автоочистка логов раз в 10 запусков

**Что сказать:**
- Проект хорошо трассируется.
- Можно посмотреть не только финальный результат, но и весь путь: какой агент работал, какой prompt ушёл, какой шаблон выбрался, где сломалась проверка.
- Для отладки LLM-систем это критически важно.
