Проблематика:
Во многих интеграционных и безопасностных контурах нельзя использовать внешние AI-сервисы для генерации кода: это создает риски утечки данных, зависимости от вендоров и нарушения требований к инфраструктуре. При этом большие модели требуют дорогих ресурсов и плохо подходят для локального развертывания. Нужна агентская система, которая работает на собственной инфраструктуре, использует легковесную локальную модель, понимает запрос пользователя на естественном языке и помогает создавать корректный Lua-код без отправки данных наружу.
Полное описание:
Участникам необходимо создать AI-агента или агентскую систему, которая локально, на базе легкой open-source LLM, принимает задачу на естественном языке и генерирует работоспособный Lua-код. При этом ценность решения не ограничивается единичной генерацией: хорошая система должна уметь задавать уточняющие вопросы, понимать контекст задачи, проверять корректность кода и помогать пользователю итеративно улучшать результат.
Для кейса важны все следующие аспекты: локальный запуск; отсутствие внешних LLM-вендоров в runtime; понятный pipeline генерации и валидации; пригодность решения для практического применения; работа в среде, где исходные данные и код не покидают контур компании. Все перечисленное существенно и учитывается при оценке решения.
Хорошим результатом будет агентская сеть, на вход которой пользователь формулирует задачу на русском или английском языке, получает первоначальную версию Lua-скрипта, затем система либо сама валидирует ответ, либо предлагает уточнения, либо запускает дополнительные шаги исправления. Дополнительным преимуществом станет использование знаний о домене или шаблонах задач, если это сделано локально и воспроизводимо.
Тех ограничения:
•	Модель должна быть легковесной; Для демо и проверки модель должна запускаться локально через Ollama на GPU 8 GB VRAM и полностью выполняться на GPU (без CPU offload). Проверка проводится на эталонном запросе организаторов при фиксированных параметрах: num_ctx=4096, num_predict=256, batch=1, parallel=1. Ограничение: пиковое потребление VRAM ≤ 8.0 GB, измеряется организаторами через nvidia-smi (peak memory). Квантизация разрешена. В README команда указывает точный ollama pull <tag> и эти параметры запуска.
•	Генерация должна работать на локальной open-source модели без обращения к внешним AI-вендорам.
•	Использование OpenAI, Anthropic и аналогичных внешних AI API для генерации или доработки кода запрещено.
•	Рекомендуемый стек: Python + open-source LLM.
•	Система должна уметь понимать задачу на естественном языке, генерировать Lua-код, вести хотя бы одну итерацию уточнения или доработки и выполнять валидацию результата в любом проверяемом виде: синтаксическая проверка, тесты, статический анализ или шаблонные проверки.
•	Все зависимости, модели и шаги запуска должны быть описаны так, чтобы жюри могло воспроизвести демо-контур.
•	Если используется база знаний, retrieval или набор шаблонов, они должны работать локально и быть включены в поставку решения и описаны в инструкции.
Критерии оценки:
1.	Качество и корректность генерируемого Lua-кода (0-50 баллов):
•	Проверяется на закрытой выборке задач, насколько код соответствует задаче, синтаксически корректен, логически последователен и пригоден к использованию.
•	Максимальный балл получают решения, в которых код не требует значительной ручной переписки после генерации.
2.	Агентность и качество итераций (0-25 баллов):
•	Проверяется, умеет ли система задавать уточняющие вопросы, воспринимать обратную связь и улучшать результат.
•	Высоко оцениваются решения, где генерация - это не один ответ модели, а управляемый цикл работы с пользователем.
3.	Локальность, приватность и воспроизводимость (0-25 баллов):
•	Проверяется, действительно ли решение можно развернуть и использовать локально без зависимости от внешних AI-сервисов, ресурсоемко.
