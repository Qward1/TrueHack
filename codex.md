# Codex instructions for this repository

Работай только от текущего состояния репозитория.
Не считай прошлую переписку источником истины.
Источник истины — код, документы проекта и git history.

## Перед началом любой задачи обязательно

Прочитай:

1. docs/AGENT_CONTEXT.md

3. docs/DECISIONS.md

## Контекст проекта

Это проект для кейса MTS True Tech Hack / LocalScript.

Цель: локальная агентская система на open-source LLM, которая:

- принимает задачу на естественном языке;
- генерирует рабочий Lua-код;
- умеет задавать уточнения или делать хотя бы одну итерацию доработки;
- валидирует результат.

## Жесткие ограничения

- Только локальная open-source модель.
- Без зависимости от внешних AI API в runtime.
- Запуск через Ollama.
- Ограничение: GPU 8 GB VRAM, без CPU offload.
- Контрольные параметры: num_ctx=4096, num_predict=256, batch=1, parallel=1.
- Все шаги запуска и зависимости должны быть воспроизводимы из README.
- Любой retrieval / knowledge base / шаблоны должны быть локальными.

## Что важно в проекте

Главное:

1. качество и корректность Lua-кода;
2. агентный цикл с уточнением и улучшением;
3. проверяемая валидация;
4. воспроизводимость локального запуска.

UI не является главным приоритетом.
Не оптимизируй решение под внешний блеск в ущерб качеству и проверяемости.

## Жесткие правила по коду

- Не создавай второй pipeline рядом с первым.
- Не оставляй старый код после замены.
- Не добавляй новые сущности без проверки существующих аналогов.
- Старайся расширять текущую архитектуру, а не плодить параллельные реализации.
- При любом существенном изменении обновляй документацию состояния.

## Обязательный протокол работы

На старте каждой задачи:

1. Кратко опиши текущее состояние.
2. Назови модули и файлы, которые уже реализуют близкую функциональность.
3. Предложи минимальный план.
4. Затем выполняй изменения.

После изменений:

1. Проверь, что старая логика не конфликтует с новой.
2. Удали мертвый код и устаревшие ветки выполнения.
3. Обнови тесты/проверки.
4. Обнови docs/PROJECT_STATE.md.
5. Если архитектура изменилась — обнови docs/DECISIONS.md.

## Чего нельзя делать

- Нельзя опираться на несуществующие функции или предполагать структуру проекта без проверки.
- Нельзя оставлять дубли в генерации, валидации, API или orchestration.
- Нельзя делать “быстрые хаки”, если они ломают воспроизводимость проекта.

## Формат итогового отчета

Всегда возвращай:

- список измененных файлов;
- краткое описание изменений;
- список проверок;
- замеченные риски;
- следующий рекомендуемый шаг.

## Definition of done

Готово только если:

- решение согласовано с текущей архитектурой;
- нет дублирующей логики;
- tests/checks обновлены;
- состояние проекта зафиксировано в docs/PROJECT_STATE.md.
